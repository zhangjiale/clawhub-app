## 1. 背景与目标

### 1.1 背景

OpenClaw是2026年最火的开源AI Agent平台，用户通过"养虾"的方式部署和管理个人AI智能体。当前大量用户同时在多个设备上运行多个OpenClaw实例，每只"虾"承担不同专业角色（产品虾、代码虾、英语虾等）。

用户与虾交互的主要入口是微信 + ClawBot插件，但微信的产品形态是为人-人通信设计的，一个微信账号只能绑定一个OpenClaw Agent。当用户养了多只虾、甚至虾分布在多个OpenClaw实例上时，通过微信管理变得极其繁琐——需要切换微信号、手动改配置、或在不同设备间来回跳转。

现有的生态工具（ClawPanel偏运维管理、openclaw-bridge偏技术中继）都不解决"移动端、个人用户、多Agent对话体验"这个核心问题。

如果不做这个产品：用户继续在微信里反复切换，多虾管理的混乱会严重降低AI助手的使用效率，"养虾"的乐趣也会被消磨。

如果做了这个产品：用户拥有一个统一的移动端入口，随时随地与任意虾流畅对话，多实例管理变得透明无感，真正释放多Agent协作的生产力。

### 1.2 目标用户

| 用户角色 | 特征描述 | 核心诉求 | 使用场景 |
|---------|---------|---------|---------|
| 个人效率爱好者 | 已在多个设备上部署了2-10个OpenClaw实例，每只虾有明确分工，日常通过微信与虾交互 | 在一个移动端界面里统一管理所有虾，随时切换对话，不想再折腾微信配置 | 通勤路上用手机让代码虾查Bug、午休时让产品虾帮忙写文案、晚上让英语虾练习口语 |
| 养虾发烧友 | 深度定制虾的Prompt和工具链，享受"养虾"过程的成就感，乐于尝试新虾配置 | 直观了解每只虾的运行状态，方便管理和调优 | 查看各虾的任务完成情况，对比不同配置的效果，快速添加新养的虾 |
| OpenClaw技术探索者 | 刚开始接触OpenClaw，正在搭建自己的第一只虾，对多实例架构感兴趣 | 降低多实例管理的门槛，有一个友好的入口来体验多Agent能力 | 在教程指导下添加多个虾实例，体验不同Agent之间的差异化交互 |

## 2. 需求概述

做一个移动端客户端（PWA或原生App），通过WebSocket直连OpenClaw Gateway，在统一界面里管理多个OpenClaw实例的多个Agent，提供流畅的多虾对话切换体验。

App采用底部三Tab导航架构：「虾列表」以实例分组方式浏览所有Agent、「消息」以微信式聊天列表查看最近对话、「实例」管理Gateway连接配置。

## 3. 功能详细设计

### 3.1 实例连接管理

**功能描述**：用户可以添加、编辑、删除和测试多个OpenClaw Gateway实例的连接配置。

**用户故事**：作为养虾用户，我希望通过扫码或手动输入的方式快速添加我的OpenClaw实例，以便不用在每个实例上反复配置。

**优先级**：P0

**业务规则**：

1. 添加实例支持两种方式：扫描OpenClaw实例生成的配置二维码（包含Gateway URL + Token），或手动输入Gateway地址（含协议、域名/IP、端口）和访问Token。
2. 每个实例配置包含以下字段：实例名称（用户自定义，如"我的MacBook"）、Gateway URL（如 `ws://192.168.1.100:18789`）、访问Token、连接状态（在线/离线/连接中）、最后连接时间。
3. 实例名称不允许为空，不允许重复。Gateway URL必须通过格式校验（ws:// 或 wss:// 开头，包含端口号）。
4. 添加实例后自动执行一次连通性测试。测试成功则标记为"在线"，测试失败则标记为"离线"并展示具体错误原因（连接超时/认证失败/地址不可达）。
5. 实例列表按最后连接时间降序排列。离线实例灰色显示，但不自动删除。
6. 删除实例前必须二次确认。删除实例同时删除该实例下所有本地缓存的对话记录。
7. Token以加密形式存储在设备本地安全存储中（iOS Keychain / Android Keystore），不以明文形式出现在界面上。
8. 支持编辑已有实例的名称、URL和Token。编辑后自动重新测试连通性。
9. 应用启动时对所有已保存实例执行一次静默连通性检测，更新各实例状态。

**交互流程**：

1. 用户从"实例管理"页面进入，看到已添加的所有实例列表。
2. 点击"+"按钮，底部弹出选择面板：「扫码添加」和「手动添加」。
3. 选择「扫码添加」→ 打开摄像头扫描界面 → 对准OpenClaw实例的配置二维码 → 自动解析并填入URL和Token → 用户确认实例名称 → 点击「连接测试」。
4. 选择「手动添加」→ 进入表单页面 → 填写实例名称、Gateway URL、Token → 点击「连接测试」。
5. 连接测试成功：显示绿色"连接成功"提示，实例添加到列表，自动拉取该实例下的Agent列表。
6. 连接测试失败：显示红色错误提示，包含具体原因（"连接超时，请检查地址和网络"/"认证失败，请检查Token"/"地址不可达，请检查实例是否启动"），用户可修改后重试。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 二维码格式不合法 | 解析失败后提示 | "无法识别的二维码，请确认这是OpenClaw配置二维码" |
| Gateway URL格式错误 | 实时校验，输入框标红 | "请输入正确的地址格式，如 ws://192.168.1.100:18789" |
| 连接超时（10秒无响应） | 终止连接，标记离线 | "连接超时，请确认OpenClaw实例已启动且在网络可达范围内" |
| Token认证失败 | 连接建立但认证被拒 | "认证失败，请检查Token是否正确" |
| 网络切换（WiFi→4G） | 检测网络变化，提示用户 | "网络已切换，部分内网实例可能不可用" |
| 应用重启后Token丢失 | 从安全存储重新加载 | 若加载失败提示"安全存储异常，请重新配置实例" |

### 3.2 Agent列表与虾选择器

**功能描述**：连接OpenClaw Gateway后，自动拉取该实例下所有已配置的Agent，以可视化卡片列表展示，供用户选择要与哪只虾对话。

**用户故事**：作为养虾用户，我希望一眼看到我所有实例下的所有虾，并能快速切换到想对话的那只虾。

**优先级**：P0

**业务规则**：

1. Agent列表从Gateway拉取，展示每个Agent的名称、描述（如OpenClaw配置了的话）、关联的Session状态（活跃/空闲）。
2. 列表按"实例"分组展示。每个实例是一个分组头，下面是该实例下的所有Agent。
3. 支持两种视图模式切换：分组视图（按实例分组）和扁平视图（所有Agent混排，按最近活跃时间排序）。
4. 每个Agent卡片展示：虾头像（默认自动生成，用户可自定义）、虾名称、所属实例名称、最后活跃时间、在线状态指示灯（绿色=在线/红色=离线/灰色=未连接）。
5. 离线实例下的Agent全部显示为离线状态，不可发起对话（但可查看历史消息）。
6. 列表顶部展示状态统计栏，以圆角卡片横向排列展示关键运营数据（活跃实例数/在线虾数/总消息数），帮助用户快速了解全局状态。统计数据实时计算，发送/接收消息后自动刷新。
7. 列表支持下拉刷新，重新从Gateway拉取最新Agent列表。
8. 用户可为每个Agent设置自定义头像和昵称（仅本地显示，不影响OpenClaw配置）。
9. Agent列表数据本地缓存，打开App时先展示缓存数据，后台静默刷新。

**交互流程**：

1. 用户打开App，进入主界面，默认展示Agent列表，顶部显示状态统计栏。
2. 列表按实例分组展示，每个分组可折叠/展开。
3. 点击某个Agent卡片 → 进入该虾的对话页面。
4. 长按Agent卡片 → 弹出操作菜单：「编辑虾信息」「置顶/取消置顶」「查看虾详情」。
5. 下拉列表 → 触发刷新，重新从Gateway拉取Agent列表和状态。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| Gateway返回空Agent列表 | 显示空状态页 | "该实例暂未配置Agent，请在OpenClaw中添加Agent后刷新" |
| Agent列表拉取失败 | 展示本地缓存数据 | "无法获取最新列表，显示的是上次缓存的数据" |
| Agent数量超过50个 | 虚拟滚动，懒加载 | 无特殊提示，保持流畅滚动体验 |

### 3.3 对话聊天界面

**功能描述**：用户与选定的虾进行实时对话，消息通过WebSocket发送到OpenClaw Gateway，Gateway路由到对应Agent处理后返回结果。

**用户故事**：作为养虾用户，我希望和每只虾的对话体验和微信聊天一样流畅自然，支持文字、图片和文件发送。

**优先级**：P0

**业务规则**：

1. 对话界面顶部展示虾头像、虾名称、所属实例名称、在线状态。点击顶部区域可进入虾详情/设置页。
2. 消息类型支持：文字消息、图片消息、文件消息、代码块消息（针对代码虾等需要展示代码的Agent）、工具调用状态消息（展示Agent正在调用什么工具、执行进度）。
3. 用户发送文字消息后，立即展示用户消息气泡（右侧），同时展示"虾正在思考..."的加载状态（左侧）。
4. 收到Agent回复后，替换加载状态为Agent消息气泡（左侧）。消息支持Markdown渲染（含代码高亮、表格、列表等）。
5. 每条消息展示发送时间。同一天内的消息按时间顺序排列，跨天显示日期分隔线。
6. 对话历史本地持久化存储，每次进入对话页加载本地历史，后台增量同步Gateway的Session数据。
7. 支持消息操作：长按消息 → 弹出菜单「复制」「删除」（仅本地删除）。
8. 底部输入栏包含：文本输入框、发送按钮、"+"按钮（展开附件面板：图片、文件、拍照）。
9. 输入框支持多行输入，自动增长高度（最多显示5行），超出后可滚动。
10. 当Agent正在执行工具调用时，在消息流中实时展示"工具调用卡片"——显示工具名称、执行状态（进行中/完成/失败）、可展开查看详细输出。
11. 支持"快捷指令"功能：用户可为每个Agent预设一组常用指令（如"/status""/memory""/reset"），在输入框上方以横向可滚动标签展示，点击即发送。
12. 对话页面支持左滑返回，返回目标为用户进入对话前的来源页面（虾列表Tab或消息Tab），实现智能返回。

**交互流程**：

1. 用户从Agent列表点击一只虾 → 进入对话页面。
2. 页面加载本地历史消息（如有），同时通过WebSocket连接Gateway建立实时通道。
3. 用户在输入框输入文字，点击发送或按回车 → 消息通过WebSocket发送到Gateway → Gateway路由到对应Agent。
4. 发送后立即展示用户消息和"虾思考中"加载动画。
5. Agent处理完毕返回回复 → 页面渲染Agent消息（Markdown格式）。
6. 如果Agent执行过程中调用了工具 → 在消息流中插入工具调用卡片，实时更新状态。
7. 用户发送图片/文件 → 编码后通过WebSocket传输 → 在对话中展示缩略图/文件卡片。
8. 用户点击快捷指令标签 → 指令文本填入输入框 → 自动发送。
9. 用户左滑或点击返回 → 返回来源页面（虾列表或消息页），对话WebSocket保持连接（后台保活）。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| WebSocket连接中断 | 自动重连（指数退避，最长间隔30秒），重连期间消息暂存本地队列 | 顶部横幅"连接已断开，正在重连..." |
| 重连3次仍失败 | 停止自动重连，提示用户手动重试 | "无法连接到虾，请检查网络或实例状态。点击重试" |
| 消息发送失败 | 消息标记为"发送失败"（红色感叹号），点击可重发 | 消息气泡上显示红色感叹号 |
| Agent响应超时（60秒无回复） | 展示超时提示，用户可取消等待 | "虾思考时间较长，可能正在处理复杂任务。继续等待/取消" |
| 图片/文件传输失败 | 展示失败状态，支持重传 | "文件发送失败，点击重试" |
| Markdown渲染异常 | 降级为纯文本展示 | 无特殊提示，确保信息可读 |

### 3.4 消息页（对话列表）

**功能描述**：提供一个独立的"消息"Tab页，以微信式聊天列表展示所有有过对话的虾，按最新消息时间排序，展示最后一条消息预览，让用户快速找到并进入最近的对话。

**用户故事**：作为养虾用户，我希望有一个像微信消息列表一样的界面，按时间顺序看到所有虾的最新消息，快速进入最近的对话，而不需要在实例分组中翻找。

**优先级**：P0

**业务规则**：

1. 消息页作为底部导航的第二个Tab，与"虾列表"和"实例"并列，是App的三个主入口之一。
2. 列表展示所有有过对话记录的虾（没有对话历史的虾不显示），按最后一条消息的时间降序排列。
3. 每行展示：虾头像（含在线状态指示灯）、虾名称、最后一条消息预览（最多40字，用户发的消息前缀"你: "）、消息时间（今天显示时分，更早显示日期）。
4. 当虾有新的未读消息时，在行尾展示红色未读角标（数字）。
5. 点击任意一行 → 进入该虾的对话页面。
6. 从对话页面返回时，智能返回到用户进入对话前的来源Tab（虾列表或消息页），而非固定返回虾列表。
7. 消息页在每次进入时自动刷新列表数据（包括消息预览和排序）。
8. 列表支持下拉刷新，从各Gateway实例重新同步最新消息状态。

**交互流程**：

1. 用户点击底部导航"消息"Tab → 进入消息页。
2. 页面展示按时间排序的虾对话列表，每行有头像、名称、消息预览和时间。
3. 有未读消息的虾显示红色角标。
4. 点击某只虾 → 进入对话页面，开始聊天。
5. 对话页面点击返回 → 回到消息页（如果从消息页进入）或虾列表页（如果从虾列表进入）。
6. 在消息页下拉 → 触发全局刷新，重新从Gateway同步最新数据。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 所有虾均无对话历史 | 显示空状态引导页 | "还没有和任何虾对话过，去虾列表找一只开始聊天吧" |
| 某个实例离线，其虾的消息预览仍可展示 | 使用本地缓存的最后一条消息 | 虾名称旁标注"离线"状态，不影响列表展示 |
| 未读消息数计算异常 | 降级为不显示角标 | 无特殊提示，确保列表可正常使用 |

### 3.5 虾个性化配置

**功能描述**：用户可以为每只虾设置自定义头像、昵称、主题色和快捷指令集，让每只虾在视觉上和功能上都有独特辨识度。

**用户故事**：作为养虾用户，我希望每只虾看起来都不一样，一眼就知道在和谁对话，并且有专属的常用命令。

**优先级**：P1

**业务规则**：

1. 可配置项：虾头像（从相册选择或拍照）、虾昵称（仅本地显示）、主题色（从预设的12种颜色中选择）、快捷指令集（每个虾最多10个）。
2. 默认头像根据虾名称自动生成（取首字+随机配色），确保不同虾有基础区分度。
3. 主题色影响该虾对话页面的顶部导航栏颜色和消息气泡色调。
4. 快捷指令格式：指令文本（如"/status"）+ 指令名称（如"查看状态"）。点击指令名称即发送对应指令文本。
5. 所有个性化配置仅存储在本地，不修改OpenClaw的Agent配置。
6. 配置修改后立即生效，无需重新连接。

**交互流程**：

1. 用户在对话页面点击顶部虾信息区域，或在Agent列表长按虾卡片选择"编辑虾信息" → 进入个性化配置页。
2. 配置页展示：头像编辑区（点击更换）、昵称输入框、主题色选择器（12色网格）、快捷指令管理区。
3. 快捷指令区：展示已有指令列表，点击"+"添加新指令（输入指令文本和显示名称），支持拖拽排序和滑动删除。
4. 点击"保存" → 配置立即生效 → 返回对话页面，导航栏颜色已更新。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 头像图片过大（>5MB） | 自动压缩至合适尺寸 | "图片已自动压缩" |
| 快捷指令数量超过10个 | 禁用"+"按钮 | "每个虾最多10个快捷指令" |
| 昵称包含特殊字符 | 允许任意字符，但长度限制20字 | 超长时提示"昵称最多20个字符" |

### 3.6 离线消息缓存与断线重连

**功能描述**：网络不稳定或Gateway实例暂时不可达时，消息不丢失，重连后自动同步。

**用户故事**：作为移动用户，我希望在地铁、电梯等网络不稳定的地方，消息不会丢失，恢复网络后自动同步。

**优先级**：P1

**业务规则**：

1. 所有已发送和已接收的消息在本地SQLite数据库中持久化存储。
2. 发送消息时如果WebSocket已断开，消息存入本地"待发送队列"，标记为"待发送"状态（灰色时钟图标）。
3. WebSocket重连成功后，按时间顺序依次发送待发送队列中的消息。
4. 重连后从Gateway拉取断线期间Agent产生的新消息（通过Session消息历史API），增量合并到本地对话记录中。
5. 消息去重：每条消息携带唯一ID（由Gateway生成），本地合并时按ID去重，避免重复显示。
6. 离线期间，Agent状态变化（如从在线变为离线）在重连后同步更新。
7. 本地存储容量管理：每个Agent对话最多缓存最近1000条消息。超出时自动清理最早的消息（保留时间轴标记）。
8. 用户可手动触发"清空本地缓存"操作（在虾详情页），清空后从Gateway重新拉取。

**交互流程**：

1. 网络断开 → 顶部出现黄色横幅"网络已断开" → 已发送消息标记为"待发送"。
2. 网络恢复 → 自动重连WebSocket → 顶部横幅变为"正在同步..." → 同步完成 → 横幅消失。
3. 待发送消息自动发出 → 状态从"待发送"变为"已发送"。
4. 断线期间Agent产生的回复自动出现在对话流中，位置按实际时间排列。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 待发送队列堆积超过20条 | 提示用户消息积压 | "有20条消息等待发送，请检查网络连接" |
| 重连后消息历史API不可用 | 仅同步待发送队列，跳过远端拉取 | "部分历史消息暂时无法同步" |
| 本地存储接近上限（>5000条总计） | 自动清理最早消息，保留最近1000条/虾 | "较早的消息已自动清理以释放空间" |
| 消息ID冲突（极小概率） | 以Gateway返回的为准 | 无特殊提示 |

### 3.7 全局消息搜索

**功能描述**：在所有虾的对话历史中统一搜索关键词，快速定位信息。

**用户故事**：作为养虾用户，我有时候忘了某条信息是哪只虾告诉我的，希望一次性搜索所有对话找到它。

**优先级**：P2

**业务规则**：

1. 搜索范围：所有已缓存的本地对话记录（所有Agent的所有消息）。
2. 搜索方式：全文关键词匹配，支持中英文，不区分大小写。
3. 搜索结果展示：匹配的消息列表，每条结果包含消息内容（关键词高亮）、所属虾名称、发送时间。
4. 点击搜索结果 → 跳转到对应虾的对话页面，并滚动定位到该消息位置（上下文展示前后各5条消息）。
5. 搜索结果按时间倒序排列（最新的在前）。
6. 搜索在本地数据库执行，不发送网络请求。

**交互流程**：

1. 用户在Agent列表页面点击搜索图标 → 进入搜索页面。
2. 输入关键词 → 实时展示匹配结果（防抖300ms）。
3. 点击某条结果 → 跳转到对应虾的对话页，目标消息高亮显示。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 无匹配结果 | 展示空状态 | "没有找到包含「xxx」的消息" |
| 搜索结果过多（>200条） | 分页加载，每页20条 | 底部"加载更多"按钮 |

### 3.8 推送通知与状态提醒

**功能描述**：当虾完成了长时间运行的任务或遇到需要用户确认的情况时，通过系统推送通知提醒用户。

**用户故事**：作为养虾用户，我希望虾完成重要任务后能主动通知我，而不是我需要反复打开App去查看。

**优先级**：P2

**业务规则**：

1. 通知触发条件（用户可在设置中逐项开关）：Agent完成任务并产生回复时、Agent执行出错时、实例连接状态变化时（在线→离线或离线→在线）。
2. 通知内容包含：虾名称、消息摘要（最多50字）、所属实例名称。
3. 点击通知 → 打开App并直接跳转到对应虾的对话页面。
4. 全局设置中可配置：通知总开关、免打扰时段（如22:00-08:00）、每个虾的独立通知开关。
5. 通知通过设备本地推送实现（不依赖远程推送服务器），由App后台服务监听WebSocket事件触发。

**交互流程**：

1. 用户在"设置 → 通知"中配置通知偏好。
2. App在后台保持WebSocket连接，监听Agent消息事件。
3. 收到触发条件的消息 → 生成系统推送通知。
4. 用户点击通知 → App打开/从后台唤起 → 直接跳转到对应虾的对话页。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 后台WebSocket被系统杀死 | 下次打开App时同步新消息 | 无推送，但打开App后消息不丢失 |
| 通知权限被拒绝 | 设置页面提示 | "通知权限未开启，请在系统设置中允许" |
| 免打扰时段内收到通知 | 静默存储，免打扰结束后批量推送 | 免打扰结束后收到"你有N条新消息"汇总通知 |

### 3.9 虾成长面板

**功能描述**：为每只虾展示一个"成长面板"，统计该虾的任务完成量、工具调用次数、活跃天数等指标，增强养虾的成就感和可视化。

**用户故事**：作为养虾发烧友，我希望直观看到每只虾的"成长轨迹"，了解自己投入了多少、虾帮了我多少忙。

**优先级**：P2

**业务规则**：

1. 统计指标：总对话次数、总消息数（发送+接收）、工具调用总次数、活跃天数（有对话的天数）、连续活跃天数、首次对话日期。
2. 数据来源：从本地对话记录中统计，不需要Gateway提供额外API。
3. 面板布局：顶部展示核心数据卡片（总对话数、活跃天数、连续天数），下方按时间线展示每日活动摘要。
4. 里程碑成就：预设一组里程碑（如"首次对话""100条消息""连续活跃7天""工具调用50次"），达成时展示庆祝动画。
5. 成就列表页展示所有已解锁和未解锁的成就，未解锁的灰色显示并标注解锁条件。

**交互流程**：

1. 用户从Agent列表长按虾卡片选择"查看虾详情"，或在对话页点击顶部虾信息 → 进入虾详情页。
2. 默认展示成长面板Tab，上方数据卡片+下方时间线。
3. 切换到"成就"Tab → 展示成就列表。
4. 达成新成就时，对话页面弹出轻量庆祝动画（3秒自动消失）。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 新安装的App无历史数据 | 从当天开始统计 | "刚开始养虾，快去对话吧！" |
| 清空对话缓存后 | 统计数据同步清零 | "统计数据已随对话记录清除" |

## 8. 风险与依赖

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| OpenClaw Gateway API不稳定，版本间有breaking change | 中 | 高 | 设计API适配层隔离变化；紧跟OpenClaw官方更新；在GitHub订阅release通知 |
| 用户OpenClaw实例在内网，手机在外网无法直连 | 高 | 高 | MVP提供内网直连方案；V1.1集成openclaw-bridge中继方案；文档中指导用户配置端口转发或Tailscale组网 |
| iOS后台WebSocket保活被系统限制 | 高 | 中 | 采用Background Task机制 + 合理心跳间隔；降级方案：下次打开App时同步；考虑接入APNs做远程推送 |
| 移动端WebSocket长连接耗电 | 中 | 中 | 动态调整心跳间隔（前台15秒/后台60秒）；低电量模式暂停保活；使用wss://减少握手开销 |
| OpenClaw社区已有类似工具（ClawPanel移动版） | 低 | 中 | 差异化定位：个人用户对话体验 vs 运维管理面板；强化"养虾"养成属性和移动原生体验 |
| 个人开发者资源有限，9周排期可能延长 | 高 | 中 | 严格MVP先行（4周），验证核心假设后再迭代；考虑PWA方案降低开发成本（省去应用商店审核） |
| 用户安全意识顾虑（Token存储在手机上） | 中 | 中 | Token加密存储（Keychain/Keystore）；支持生物识别解锁；开源代码接受社区审计；文档说明安全架构 |

### B. OpenClaw Gateway WebSocket协议参考

连接地址：`ws(s)://<host>:18789`

请求格式：
```json
{
  "type": "req",
  "id": "<unique-request-id>",
  "method": "<method-name>",
  "params": { }
}
```

认证方式：Token + 设备ID + 配对码（三层认证）

### C. 竞品参考

| 产品 | 定位 | 与ClawHub的差异 |
|------|------|-------------|
| ClawPanel | OpenClaw可视化管理面板 | Web端运维导向，无移动原生体验，无对话优化 |
| 微信+ClawBot | 通过微信与Agent对话 | 单Agent绑定，无法管理多实例，无虾切换 |
| openclaw-bridge | WebSocket中继服务 | 纯技术工具，无UI，不解决对话体验 |
| ChatBox/NextChat | 多AI模型统一客户端 | 面向通用LLM，不支持OpenClaw Agent协议 |
