

### 去触发器化（Trigger-less）的现代重构方案

我们将原本隐藏在数据库底层的逻辑，**全部上移到 Data Layer 的 Repository 层**，通过**应用层事务（Application-Level Transaction）** 来保证一致性。这样既能让 ORM 完美追踪变更，又能让逻辑清晰可见、易于测试。

#### 1. 重构：1000 条红线淘汰机制
**放弃每次 Insert 都执行 `COUNT(*)` 的 Trigger**，改为在 Repository 层采用 **“阈值清理策略”** 或 **“定期清理策略”**。

* **优化方案**：在 `MessageRepository` 中维护一个内存计数器，或者在每次插入消息后，异步检查该 Agent 的消息总数。为了极致性能，我们不在每次插入时清理，而是**允许短暂超限（如达到 1050 条时），一次性清理掉最旧的 100 条**，大幅减少数据库 I/O。

```dart
// Data Layer: MessageRepository.dart
class MessageRepository {
  // 内存缓存每个 Agent 的当前消息数，避免频繁 COUNT(*)
  final Map<String, int> _agentMsgCounts = {};

  Future<void> insertMessage(Message msg) async {
    await db.transaction(() async {
      // 1. 插入消息 (ORM 能感知，UI 会刷新)
      await db.messages.insertOne(msg);
      
      // 2. 应用层双写 FTS5 索引 (ORM 能感知)
      await _syncFts5Index(msg);
      
      // 3. 应用层阈值清理 (替代 Trigger)
      await _enforceMessageLimit(msg.agentId);
    });
  }

  Future<void> _enforceMessageLimit(String agentId) async {
    // 懒加载计数
    _agentMsgCounts[agentId] ??= await db.messages.countByAgent(agentId);
    _agentMsgCounts[agentId] = _agentMsgCounts[agentId]! + 1;

    // 阈值触发：超过 1050 条时，清理掉最旧的 100 条，留出 Buffer
    if (_agentMsgCounts[agentId]! > 1050) {
      final oldestIds = await db.messages
          .selectOnly()
          .where(db.messages.agentId.equals(agentId))
          .orderBy(db.messages.timestamp.desc())
          .limit(100, offset: 950) // 保留最新的 950 条
          .map((row) => row.read(db.messages.clientId)!)
          .get();
          
      await db.messages.deleteWhere((m) => m.clientId.isIn(oldestIds));
      _agentMsgCounts[agentId] = 950; // 重置计数
    }
  }
}
```

#### 2. 重构：FTS5 全文索引同步
**放弃 `AFTER INSERT/UPDATE/DELETE` Trigger**，改为在应用层的同一个事务中，显式调用 FTS5 的插入/删除命令。

* **优化方案**：利用 SQLite FTS5 的 **External Content Table（外部内容表）** 特性，但在应用层手动维护同步。

```dart
// Data Layer: SearchRepository.dart
class SearchRepository {
  // 在 MessageRepository 的 insert/update/delete 事务中被调用
  Future<void> syncFtsIndex(Message msg, {bool isDelete = false}) async {
    if (isDelete) {
      // 使用 FTS5 的特殊 delete 命令
      await db.customUpdate(
        'INSERT INTO messages_fts(messages_fts, rowid, content) VALUES(\'delete\', ?, ?)',
        variables: [Variable.withInt(msg.rowid), Variable.withString(msg.content)],
      );
    } else {
      // 显式插入索引
      await db.customInsert(
        'INSERT INTO messages_fts(rowid, content, agent_name) VALUES (?, ?, ?)',
        variables: [
          Variable.withInt(msg.rowid), 
          Variable.withString(msg.content),
          Variable.withString(msg.agentName)
        ],
      );
    }
  }
}
```

#### 3. 重构：Schema 清理（彻底移除 Trigger）
在数据库的 DDL（数据定义语言）中，**彻底删除所有 `CREATE TRIGGER` 语句**。数据库 Schema 变得极其干净，只保留纯粹的表结构定义。

```sql
-- 干净的 messages 表，没有任何 Trigger 负担
CREATE TABLE messages (
    rowid INTEGER PRIMARY KEY AUTOINCREMENT,
    client_id TEXT UNIQUE NOT NULL,
    server_id TEXT UNIQUE,
    agent_id TEXT NOT NULL,
    content TEXT,
    timestamp INTEGER NOT NULL
    -- ... 其他字段
);
CREATE INDEX idx_msgs_agent_time ON messages(agent_id, timestamp DESC);

-- 干净的 FTS5 表 (External Content Table)
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,
    agent_name,
    content='messages',
    content_rowid='rowid',
    tokenize='unicode61'
);
-- 注意：这里不再有任何 AFTER INSERT/UPDATE/DELETE 触发器！
```
