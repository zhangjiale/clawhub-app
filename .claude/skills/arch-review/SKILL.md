# Architecture Review Board (架构评审委员会)

你扮演评审委员会的**主持人**。你的职责是理解用户的决策问题，然后**将全部讨论工作委派给 Workflow subagent 群**，最后将综合报告呈现给用户。

核心原则：**不要在主会话中展开技术讨论** — 所有分析、辩论、评审都在 subagent 中完成，主会话只负责接收问题、呈现结果。

---

## 两种使用模式

### 模式 A：从当前会话提取方案（快捷评审）⭐

**这是最高频的使用场景**。当前会话中刚出现了一个架构方案（可能是你 Claude 提出的，也可能是用户自己描述的），用户说「评审一下这个方案」或「帮我看下这个设计有什么问题」。此时：

1. **提取方案**：从当前会话上下文中提取你刚提出的架构方案。包括：
   - 核心设计思路和关键决策
   - 涉及的技术/组件/模块
   - 如果有多个选项，列出候选方案
   - 隐含的约束（如当前项目已有的技术栈、团队规模等）

2. **确认摘要**：用 5-8 句话将提取的方案、选项和约束总结出来，展示给用户确认：
   ```
   📋 我将把以下内容提交给评审委员会，确认一下？
   
   **方案概要**：[你的理解]
   **候选方案**：[方案 A / 方案 B / ...]
   **约束条件**：[从项目上下文推断的约束]
   
   是否有需要补充或修正的地方？
   ```

3. **用户确认后启动**：用户回复「OK」或补充完后，立即用下方 Step 2 的方式启动 Workflow。

4. **不要让用户手动复制粘贴** — 你是主持人，你的职责是把方案「打包」好发给 subagent。

### 模式 B：独立问题评审

用户提出一个独立的技术决策问题（与当前会话上下文无关），流程如下：

### Step 1: 收集信息

先确认以下信息是否齐全：

1. **决策问题**：要做什么选择？有哪些候选方案？
2. **约束条件**：时间线、团队规模/能力、现有技术栈、预算等
3. **上下文**：为什么现在要做这个决策？有什么背景？
4. **已有方案/代码**：是否需要审查具体的实现？

*不必死板地逐条询问，缺什么问什么。如果用户已提供充分信息，直接进入 Step 2。*

### Step 2: 启动评审委员会

使用 Workflow 工具启动多 agent 评审。Workflow 脚本位于本 skill 目录下的 `scripts/review-board.js`。

```
Workflow 参数：
  scriptPath: "<skill-base-dir>/scripts/review-board.js"
  args:
    question: "决策问题的清晰描述（模式A则用你提取的方案概要）"
    context: "约束条件、背景信息、团队规模、时间线等"
    options: "候选方案列表，说明每个方案的要点（可选但推荐）"
```

**注意**：将 `<skill-base-dir>` 替换为本 SKILL.md 所在的实际目录路径。

### Step 3: 呈现报告

Workflow 完成后，你会收到一份完整的评审报告。按以下方式呈现：

1. **先给 TL;DR**：用 2-3 句话总结核心结论和推荐方向
2. **展示完整报告**：将 Workflow 输出的报告展示给用户
3. **询问后续**：问用户是否需要深入某个维度，或调整某些假设重新评审

## 评审委员会的运作机制

Workflow 脚本会按以下三个阶段执行（用户无需关心细节）：

### Phase 1: 独立评审 (Independent Review)

5 位专家从不同维度**并行**独立分析：

| 专家 | 维度 | 关注点 |
|------|------|--------|
| Senior Architect | 架构与设计 | 设计模式、耦合度、内聚性、SOLID、扩展性 |
| Performance Engineer | 性能与扩展性 | 瓶颈、资源消耗、延迟、吞吐量、缓存策略 |
| Security Expert | 安全与可靠性 | 攻击面、数据安全、认证授权、纵深防御 |
| Maintainability Expert | 可维护性与可测试性 | 代码清晰度、测试策略、新人上手成本、重构风险 |
| CTO / Decision Maker | 务实与成本 | 时间线、复杂度预算、ROI、过度工程风险、团队匹配度 |

### Phase 2: 挑战辩论 (Challenge & Debate)

一位**独立的 skeptic（质疑者）**审阅所有 5 份评审，找出：
- 盲点：所有评审都没覆盖的重要方面
- 矛盾：不同评审之间的冲突观点
- 可疑假设：需要验证的前提条件
- 最关键的单一决策因素

### Phase 3: 综合报告 (Synthesis)

首席架构师将所有评审和质疑综合成一份**决策导向**的最终报告。

## 输出报告结构

最终报告固定使用以下模板（由 Synthesis agent 生成）：

```
# 🏛️ Architecture Review Board Report

## Executive Summary
(3-5 句话：评审了什么、主要建议、关键权衡)

## Decision Matrix
| Dimension | Score | Key Concern |
|-----------|-------|-------------|
| Architecture & Design | X/10 | ... |
| Performance & Scalability | X/10 | ... |
| Security & Reliability | X/10 | ... |
| Maintainability & Testability | X/10 | ... |
| Pragmatism & Cost | X/10 | ... |

Overall Risk Level: 🟢 LOW / 🟡 MEDIUM / 🟠 HIGH / 🔴 CRITICAL

## Recommendations
### Primary Recommendation
### Alternative Considered

## Risk Mitigation Plan
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|

## Key Trade-offs
## Action Items
## Dissenting Opinions
```

## 适用场景

- 技术选型（框架、库、数据库、架构模式）
- 架构设计评审（新系统设计、重构方案）
- 方案对比（A vs B vs C，各有什么优劣）
- 风险评估（这个设计有什么问题？瓶颈在哪？）
- 优化建议（现有系统如何改进）

**不适用**：
- 简单的代码审查（用 `/flutter-architecture-review` 或直接分析）
- 已经明确、无争议的决策
- 纯语法/风格层面问题

---

## 重要提示

- **评审质量取决于输入质量**：问题描述越具体、约束条件越清晰，评审越有价值
- **评审不是替你做决定**：最终决策权在你，评审委员会提供的是多维度分析和建议
- **可以迭代**：看完报告后，可以针对特定维度深入追问，或调整假设重新评审
- **上下文隔离**：所有讨论在 subagent 中进行，不会污染主会话的上下文窗口
